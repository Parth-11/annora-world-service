1.  Worlds  as  Structured  Lore  Graphs  
A  world  is  just:  
●  Places  ●  Species  ●  Artifacts  ●  Factions  ●  Events  
Each  is  a  node  with  a  variable  schema  (certain  parameters  are  required,  such  as  type,  
name,
 
etc.)
.
 
Example:  Flora  Entry  {    "type":  "flora",    "name":  "Silverbloom",    "traits":  ["glows  at  night",  "poisonous  pollen"],    "native_to":  ["Moonvale"],    "created_by":  "user_id",    "canon_status":  "accepted"  }   Other  custom  parameters  can  be  referenced  under  events  or  while  writing  the  wiki.    
2.  Canon  Is  Decided  by  Voting  
Flow  
1.  A  member  proposes  a  new  entry  2.  It  enters  “Pending  Lore”  3.  Community  votes:  ○  Accept  into  canon  ○  Reject  ○  Send  for  revision  
Rules  
●  Voting  weight  =  reputation  ●  Low-rep  users  can  still  propose  

●  Nothing  is  permanent  instantly  
This  replaces:  
●  Editors  ●  Mods  ●  Lore  masters  
 
3.  “Exploration”  Without  Simulation  
Instead  of  someone  walking  around  the  world:  
Users  do:  
●  Browse  interconnected  lore  nodes  ●  Follow  links:  ○  City  →  Creature  →  Event  →  Artifact  ●  Discover  stories  through  relationships  
This  is  closer  to:  
●  Tolkien  appendices  ●  Wiki  rabbit  holes  ●  Dark  Souls  environmental  storytelling  
No  AI  pathfinding.  No  movement.  
 
4.  Events  as  Story  Anchors  (Not  Quests)  
Events  are  historical  moments ,  not  live  gameplay.  
Example  
Event:  The  Fall  of  Moonvale  
●  Date  (fictional)  ●  Involved  factions  ●  Consequences  ●  Linked  places  &  creatures  
Events:  

●  Can  be  proposed  ●  Voted  into  canon  ●  Referenced  by  future  entries  
This  creates  timeline  depth  without  mechanics.  
 
5.  Storytelling  via  “Story  Threads”  
Instead  of  roleplay  or  AI  narration:  
Story  Thread  
●  A  constrained  storytelling  format:  ○  1  paragraph  per  user  ○  Max  500  chars  ○  Must  reference  existing  canon  nodes  
Backend  enforcement  
●  Users  must  select:  ○  Location  ○  Event  context  ○  Characters  involved  
This  prevents  chaos  and  keeps  consistency.  
 
6.  Light  Quests  =  Writing  Prompts  
Quests  are  prompts ,  not  missions.  
Examples:  
●  “Write  a  story  involving  a  forbidden  place”  ●  “Describe  a  creature  feared  by  sailors”  ●  “Create  an  artifact  tied  to  an  ancient  event”  
Completion:  
●  Unlocks  cosmetic  badges  ●  Gives  reputation  

●  Adds  lore  entries  
 
8.  Minimal  AI  Usage  (Optional,  Safe,  Cheap)  
AI  can  be  used  only  for:  
●  Grammar  cleanup  (optional)  ●  Auto-linking  related  entries  ●  Summarizing  seasons  
AI  NEVER:  
●  Decides  canon  ●  Creates  lore  ●  Plays  characters

 Service  Name  Language  Explanation  of  Choice  
Auth  Service  Golang  Golang  is  ideal  here  due  to  its  strong  security  features,  fast  execution  for  token  generation/validation,  and  low  overhead  for  handling  authentication  requests,  which  are  critical  and  frequent.  It's  statically  typed,  reducing  bugs  in  security-sensitive  code,  and  performs  well  under  load  compared  to  JS,  which  could  introduce  more  runtime  errors.  

User  Service  JavaScript  (Node.js)  
JS  is  chosen  for  its  async  nature,  perfect  for  handling  real-time  reputation  updates  and  badge  unlocks  triggered  by  events.  The  service  involves  relational  data  and  frequent  queries  (e.g.,  weighting  votes),  where  Node's  non-blocking  I/O  excels.  Its  ecosystem  (e.g.,  libraries  like  Mongoose  for  MongoDB,  if  used)  enables  quick  prototyping  of  user  profiles,  outweighing  Go's  performance  since  this  isn't  compute-heavy.  
World  Service  Golang  This  service  manages  hierarchical  metadata  and  timelines,  benefiting  from  Golang's  concurrency  for  efficient  world  joins/memberships  queries.  It's  data-oriented  with  potential  for  complex  aggregations  (e.g.,  summaries),  where  Golang's  speed  and  memory  efficiency  shine  over  JS,  especially  for  long-running  apps.  
Lore  Service  Golang  As  the  core  graph-handling  service  (CRUD,  linking,  validation),  Golang  is  a  natural  fit  for  performance-critical  graph  operations  (e.g.,  with  Neo4j  drivers).  Its  goroutines  handle  concurrent  node  edits/searches  efficiently,  and  static  typing  ensures  schema  integrity.  JS  would  be  slower  for  large  graphs,  making  Golang  better  for  scalability.  
Vote  Service  Golang  Reconsidered  from  JS:  Voting  involves  time-sensitive  tallies  with  reputation  weighting,  which  can  spike  during  events.  Golang's  concurrency  and  math  performance  make  it  superior  for  accurate,  fast  computations  under  load.  While  JS  handles  async  well,  Golang  avoids  callback  hell  and  provides  better  reliability  for  dispute-free  outcomes.  
Event  Service  Golang  Events  require  precise  scheduling  (e.g.,  cron-like  phase  transitions)  and  orchestration  across  services.  Golang's  standard  library  for  time  management  and  workflows  (e.g.,  Temporal  integration)  makes  it  reliable  for  automated,  time-boxed  processes.  JS  could  work  but  risks  timing  inconsistencies  in  high-concurrency  scenarios.  
Story  Service  JavaScript  (Node.js)  
This  collaborative  service  benefits  from  JS's  event-driven  model  for  real-time  thread  contributions  (e.g.,  via  Socket.io).  Validating  references  and  enforcing  constraints  (e.g.,  char  limits)  is  async-heavy,  where  Node  excels.  Golang  could  do  it,  but  JS's  flexibility  speeds  up  development  for  narrative  features.  

Notification  Service  
JavaScript  (Node.js)  
JS  is  perfect  for  pub/sub  patterns  and  async  delivery  (e.g.,  with  Redis  or  RabbitMQ).  It  handles  real-time  pushes  efficiently,  and  libraries  like  Nodemailer  or  Firebase  integrate  seamlessly.  Golang  could  be  used,  but  JS's  ecosystem  makes  it  faster  to  implement  and  scale  for  user-facing  alerts.  
Search  &  Browse  Service  (Optional)  
Golang  If  implemented  separately  for  complex  graph  searches,  Golang's  performance  in  query  optimisation  and  traversal  algorithms  makes  it  the  choice.  It  handles  large  datasets  better  than  JS,  ensuring  responsive  "exploration"  flows  without  lag.  
AI  Service  (Optional)  
JavaScript  (Node.js)  
Minimal  AI  tasks  (e.g.,  grammar  checks  via  API  calls)  are  integration-focused,  where  JS's  vast  NPM  ecosystem  (e.g.,  OpenAI  SDK)  allows  easy,  async  handling.  It's  optional  and  low-load,  so  JS's  rapid  development  trumps  Golang's  efficiency  here.
